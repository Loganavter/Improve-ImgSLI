# Development History

This document tracks the evolution of Improve-ImgSLI across releases and major refactors. It is migrated from the former "Development Story" section of README to keep the main page concise.

## September 2024 — First prototype
The journey of Improve ImgSLI began in September 2024, initially conceived with ChatGPT's assistance to address a personal need for straightforward image comparison in my work. The early version provided basic side-by-side functionality.

## October 2024 — Magnifier and drag-and-drop
The following month, October, marked an enhancement phase using Claude, which introduced a magnifier tool and drag-and-drop support, significantly improving usability.

## November 2024 — Magnifier refinements
By November, further refinements to the magnifier were implemented with Claude, including view freezing and merging capabilities. However, the codebase's expansion, coupled with Claude's 8k token context limit, made complete code regeneration impractical, necessitating manual integration of new features. Gemini was then employed to help manage these integrations, though not all AI-generated code was directly usable, leading to some features being postponed or temporarily disabled to maintain stability.

## December 2024 — Experiments and lessons
A brief period in early December saw an experiment with adaptive magnifier positioning tied to window resizing, which was ultimately set aside due to unsatisfactory results.

## January 2025 — Dynamic resizing
A turning point came in January 2025 when a user request for dynamic window resizing led to the exploration of DeepSeek. This AI, notable at the time for its Chain-of-Thought reasoning, successfully guided the implementation of this crucial feature, with Gemini then seamlessly incorporating it alongside other updates. See issue: https://github.com/Loganavter/Improve-ImgSLI/issues/1

## February 2025 — Feature growth and stability
February 2025 brought renewed focus on enhancing the application. Leveraging Claude Sonnet 3.7, features like dynamic image swapping, a multilingual dictionary, and further magnifier improvements were added, along with a helpful tooltip. Shortly thereafter, access to Grok 3—first via X, then its dedicated website—provided a significant boost. Grok 3's DeepThink model proved exceptionally effective at resolving persistent bugs, outperforming previous AIs, while its generous query limits and intelligent standard model streamlined development. Key contributions included optimized rendering, filename display and editing, and fixes for fullscreen mode.

## Late March 2025 — Cross-platform push
Late March 2025 was dedicated to cross-platform compatibility. Gemini and Grok, using their web search capabilities, assisted in drafting build and packaging scripts. While initial hopes for community maintenance of platform builds didn't materialize due to some skepticism about the project's AI-assisted origins, this solo effort, though time-intensive (taking considerable effort per platform), led to the discovery and resolution of several bugs, culminating in a successful cross-platform launch. Delays in Flathub publication until mid-April allowed for further bug fixes and the addition of a list cleanup feature.

## End of March 2025 — Timeline completion and architectural cleanup
By the end of March 2025, after several more days of messy but productive work with Claude and Codex, the long-running timeline overhaul was finally brought to a proper stopping point. The design was at last finished, and the overall structure became solid enough to prepare the feature for future editing capabilities rather than just passive viewing. At the same time, the contract between the viewing layer and the editing layer still felt insufficiently mature, so the final transition from a viewing-oriented timeline to a full editing workflow was left for a later stage.

This same stretch of work also turned into a broader cleanup pass across the application. Wherever possible, popup-like widgets were removed and rewritten as same-window integrated widgets drawn directly inside the main interface. Roughly 30% of the architecture was revisited again over the course of several more days. QSS styles were no longer concentrated in a single shared file and were instead split across the project according to responsibility. Windows started behaving much more predictably, most of the remaining core monoliths were dismantled, MVP was brought back to a cleaner and more correct form, and some parts of the code were optimized by extracting repeated logic into reusable helpers and services.

## April 2025 — Persistence and settings
In April 2025, attention shifted back to addressing critical bugs, with continued reliance on Gemini. A major achievement during this period was the implementation of window state persistence—a feature that proved incredibly challenging but vastly improved user experience. A settings tab was also introduced, centralizing language selection, output image quality, and filename length preferences. Following these updates, Windows and Flatpak builds were refreshed. The current plan involves implementing one final feature and verifying the Flatpak maintenance pipeline before updating Windows and AUR builds, after which the project will likely enter a period of stability.

## May 2025 — Caching and architecture split
May 2025. I discovered Cursor AI, which allowed me to quickly implement three key features in this update using agents like ChatGPT 4.1, Sonnet 3.5, and Gemini 2.5 PRO, all within the free limit of 150 uses. These features included a smarter caching service that improved smoothness by approximately 2-3 times compared to the previous version, the ability to select magnifier interpolation, and quick image preview using the keyboard. Additionally, the project's structure became much more multi-layered and less coupled between modules. This refactoring required two days of collaboration with Gemini to break down two monolithic scripts into numerous services with separate folders. The entire process took about three days.

## June 2025 — UI style refactor
June 2025. With Gemini, I refactored the code once again. The application now features a qfluentwidgets style. It initially took a long time, but then I gave the project fork code with this GUI, and it went much faster, it took about 3 days. Caching was also further improved. The icons.py file was completely removed as it was no longer necessary.

## July 2025 — Rendering pipeline overhaul
July 2025. The solid architecture from the June refactoring allowed for another rapid development burst. This update took about 4-5 days, again with Gemini's assistance. The launcher.sh script was turned into a proper command-line utility for managing the environment. The rendering pipeline was overhauled to use a dynamic canvas, which allows the magnifier to draw outside the image bounds and also improves performance on large images and magnifiers due to more efficient data packaging. The magnifier also now correctly handles horizontal splitting. UI interaction was improved with long-press actions and scrollable selection dropdowns. Internally, all print statements were replaced with a standard logging system. The persistent bug with window geometry, however, remains. After over 20 failed attempts to fix it, it's clear the root cause is a race condition in how state signals are handled. This confirms the need to move away from the current semblance of an architecture to a full MVC/MVVM pattern, which will be the primary focus going forward.

## August 2025 — Full custom UI and MVP
August 2025. This update was born out of necessity. The qfluentwidgets library, while initially useful, became a major roadblock due to its extreme rigidity in customization. Discovering that simple layout logic was hardcoded in its C++ source was the final straw. This prompted a full rewrite of the user interface with 100% custom components. In parallel, the long-standing "god script" issue was tackled by migrating the entire application to a proper Model-View-Presenter (MVP) architecture. This new, solid foundation allowed for the rapid implementation of several new features: an image rating system, selectable light/dark themes, a display cache for performance with large files, and completely redesigned settings and help dialogs. The entire transformation was completed in about a week, exclusively with Gemini's help.

## Early October 2025 — Shared toolkit and UX polish
Early October 2025. With renewed access to Cursor AI, a significant unification of the codebase with another of my projects, Tkonverter, was carried out. The result is a shared library, shared_toolkit, which simplifies transferring functionality between projects. Following a user request, a feature for pasting images from the clipboard (Ctrl+V) was added. The user experience was substantially redesigned, introducing new customization options for the main comparison line and the magnifier's divider (controlling visibility, color, and thickness). System notifications on Linux were also fixed. In addition to Cursor, this update was developed with the active use of Gemini and Claude models, as well as new AI aggregator platforms, which helped accelerate the process. See issue: https://github.com/Loganavter/Improve-ImgSLI/issues/20
Further enhancements to the magnifier (precision controls and subpixel rendering for smoother comparisons between different resolutions), plus the addition of pixel/structural difference views and analysis tools (channels, edges, metrics). This batch of work took approximately 3–5 hours.

## December 2025 – January 2026 — Plugin Architecture and Video Export
This update represents a massive architectural overhaul that took nearly a week of intensive work. After experimenting with several patterns, including Redux-like state management, the application was migrated to a more flexible and scalable plugin-based architecture.

The flagship feature of this release is the introduction of a video editor and export functionality, allowing users to record their comparison sessions, perform basic trimming, and export the result to various video formats. The user experience of the main toolbar was also redesigned for clarity and better grouping of functions.

Several quality-of-life improvements were added, such as automatic cropping of black borders on image load, magnifier guide lines (or "lasers") to visually connect the magnifier to its capture point, and support for the JPEG XL format. This release was a significant effort to stabilize the new architecture and deliver a cohesive set of powerful new features. Development was primarily done using Cursor AI with models from Claude and ChatGPT, as well as separately using Gemini 3.5 Pro Preview.

## February–March 2026 — Architectural Overhaul and Hybrid Rendering
This period saw a two-part, substantial update focused on deep architectural refactoring and performance. The first part involved a complete overhaul of the canvas architecture. This long-overdue refactor resulted in a highly optimized system, with all operating modes, comparison types, and edge cases fully debugged and stabilized. The second part introduced the most significant new feature: hybrid CPU-GPU rendering. This effort also involved another deep dive into the architecture, further breaking down services into sub-services and plugins.

The hybrid rendering system was developed with the help of Claude Code and Opus 4.6 (despite their restrictive quotas). Together, we designed a transition framework for scaling the entire rendering subsystem, which included writing approximately 80% of the new shaders. The process took several days, as the legacy pipeline was heavily optimized for the CPU, leading to unexpected bugs when integrating the new GPU-based components. For more applied tasks, ChatGPT Codex and ChatGPT 5.4 were used; they proved to be much faster and highly capable, assisting with tasks other than the core rendering system.

In another major step forward, all pop-ups and separate windows—which previously mimicked being part of the main window—were rewritten as native Qt widgets with Claude's assistance. The only exception was the image list pop-up, which was meticulously redesigned to be pixel-perfect for the first time since its initial implementation in early 2025. This change brings the application significantly closer to running natively on Wayland without the X11 compatibility layer, though some nuances still need to be addressed. The conversion of most widgets was surprisingly fast, taking about an hour, but the image list required experimenting with nearly 10 different architectural approaches to find the optimal layout and rendering method.

The video editor also received a significant overhaul:
*   The preview renderer can now fill empty areas with any custom color.
*   The timeline was rewritten from the ground up to be more robust and intuitive. Thumbnails now preserve their aspect ratio and are rendered in high quality.
*   Timeline markers are now dynamically spaced based on thumbnail dimensions rather than being tied to a hardcoded 16:9 aspect ratio and video time.
*   Internally, the editor moved from a snapshot-oriented approach to a keyframe-based model, laying the groundwork for proper editing.
This work took around 2.5 hours.

A key new feature is the addition of canvas zooming and panning. This was implemented in just about 3 hours but involved complex mathematics developed with Opus and ChatGPT.

The program was also prepared for the eventual introduction of workspaces, although no new tabs are expected in the immediate updates. The longer-term plan is to extend this foundation toward video comparison, but first I still need to finish several pending improvements in the core image-comparison workflow: the video editor itself is not fully finished yet, the filename/text overlay system still needs more work, and a color picker is also planned.

In total, this series of updates took over a week of late-night coding sessions. While I'm not convinced that a CLI-first workflow is definitively superior to IDEs, the higher quotas were a significant advantage over tools like Cursor. However, Cursor excels in facilitating learning, as it allows you to visually absorb new code patterns during development. I believe that developing the UI toolkit without an IDE like it would be unrealistic. A broader distribution of everything accumulated since November 2025 is planned slightly later as version 8.0.0.

## Early April 2026 — Compatibility patching and release pipeline hardening
Early April 2026 turned into a cleanup phase after the huge February-March refactor. Most of the work here was not about adding flashy new features, but about finally chasing down the annoying edge cases that only became visible once zooming, panning, hybrid rendering, and the new backend split started interacting with each other. With Codex and ChatGPT 5.4, I spent a lot of time fixing split-line behavior near the image borders, correcting magnifier geometry and clamping, and reducing high-DPI mismatches where the visible render area and the actual interaction area did not quite agree.

Another major topic was startup behavior and backend compatibility. The program moved further toward the Quick canvas path, because the old OpenGL widget path kept exposing more platform-specific quirks than I wanted to keep fighting forever. The prewarm and reveal logic also had to be reconsidered. Right now that old startup behavior still works best on GNOME Wayland; on KDE Wayland and Windows it showed various compositor and window-lifecycle artifacts. Those fixes are not finished yet and will need a separate round of testing later, so for now this part should be considered only partially solved.

The video editor also got a small but important safety fix: trimming without an actual selection no longer deletes the whole clip. At the same time, version 8.2.0 packaging was cleaned up across Flatpak, AUR, and Windows templates, and a first proper Windows GitHub Actions pipeline was added. It now builds the PyInstaller bundle and the Inno Setup installer automatically, and before doing so it checks that release metadata is synchronized across the packaging files. This work was much less glamorous than the renderer rewrite, but it was necessary if the project was going to start behaving like a real maintained application rather than just a constantly changing codebase.
